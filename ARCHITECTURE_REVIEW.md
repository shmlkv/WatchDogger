# Ревью архитектуры WatchDogger

macOS menu bar утилита для мониторинга CPU (~890 строк Swift, без внешних зависимостей).

---

## Сильные стороны

- **Минимализм и автономность** — чистый AppKit/SwiftUI, никаких SPM-зависимостей, компилируется одной командой `swift main.swift`
- **Sustained detection** — требует 2+ последовательных превышения порога, защита от ложных срабатываний на разовых спайках
- **Battery awareness** — адаптивный порог с кешированием состояния батареи (раз в 5 циклов)
- **Грамотная работа с потоками** — мониторинг на background queue, UI на main thread
- **Kill/Ignore через notification actions** — удобный UX без открытия окна

---

## Проблемы и рекомендации

### 1. God Object — `WatchDoggerDelegate` (465 строк)

`main.swift:4-465` совмещает 5 ответственностей: lifecycle, UI, мониторинг, нотификации, меню. Стоит выделить:

- `ProcessMonitor` — логика `ps`, парсинг, sustained detection, kill
- `NotificationManager` — настройка, отправка, обработка actions
- `MenuBarController` — статус-айтем, динамическое меню

### 2. Shell-based мониторинг через `ps` — хрупкость

`main.swift:237-249` — парсинг stdout внешнего процесса зависит от формата вывода `ps`. Рекомендую перейти на `sysctl` / `proc_pidinfo` (libproc) — нативный API, не ломается между версиями macOS, быстрее spawn процесса.

### 3. Нет обработки ошибок при spawn процессов

`main.swift:148-157` — `Process()` может бросить исключение, но обработка минимальна. При недоступности `/bin/ps` (sandbox, SIP) приложение молча не работает. Нужен fallback или уведомление пользователя.

### 4. Нет блокировки повторного запуска

Ничто не мешает запустить два экземпляра — два таймера, дублирующиеся нотификации. Стоит добавить проверку через `NSRunningApplication` или file lock.

### 5. Потенциальная race condition в `cpuHistory`

`cpuHistory` и `lastNotified` модифицируются из background queue (`main.swift:287-298`), а читаются из main thread при обновлении меню. Нет явной синхронизации — в текущем виде это может дать data race. Решение: `DispatchQueue` с барьером или `@MainActor`.

### 6. Жёсткая привязка к конкретным браузерам

`monitorSafari` / `monitorChrome` — булевы флаги вместо конфигурируемого списка процессов. Масштабируется плохо. Лучше: `monitoredProcessNames: [String]` с пресетами.

### 7. Нет тестов

Для утилиты с kill-логикой (`SIGTERM`) отсутствие тестов — риск. Минимально стоит покрыть:

- Парсинг вывода `ps`
- Логику sustained detection
- Cooldown/ignore фильтрацию

### 8. `Info.plist` — bundle ID `com.sh.watchdogger3`

Суффикс `3` в идентификаторе выглядит как legacy от итераций. Если это production-версия — стоит зафиксировать `com.sh.watchdogger`.

## Итоговая оценка

| Критерий | Оценка |
|----------|--------|
| Простота | Отлично |
| Зависимости | Отлично |
| Разделение ответственностей | Слабо (god object) |
| Надёжность | Средне (ps parsing, нет error handling) |
| Thread safety | Слабо (data race) |
| Тестируемость | Отсутствует |
| Расширяемость | Средне |

**Приоритет #1** — исправить data race в `cpuHistory`. Это единственный баг, который может проявиться в runtime. Остальное — архитектурный долг, который можно закрывать постепенно.
